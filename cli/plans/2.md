Below is a rigorous product-level specification covering all five requested areas. This is written as a system design and product architecture document, not an implementation guide.

# 1. Formal CLI UX Specification

## 1.1 Command Hierarchy

```
latest
  ├── -v | --version
  ├── init
  ├── check [--compatibility] [--json] [--non-interactive]
  ├── update [package-name] [--compatibility] [--dry-run]
  ├── doctor
  └── help
```

Design principles:

* Deterministic command behavior.
* Interactive by default in TTY.
* Non-interactive in CI environments.
* Structured output available via `--json`.

---

## 1.2 Global UX Principles

### Output Philosophy

* High signal, low noise.
* Structured table layout.
* Risk-first presentation.
* Always show:

  * Current version
  * Target version
  * Risk classification
  * Compatibility status

### Color Semantics

| Color  | Meaning            | Definition                         |
| ------ | ------------------ | ---------------------------------- |
| Green  | Patch              | `x.y.z → x.y.z+1`                  |
| Yellow | Minor              | `x.y.z → x.y+1.0`                  |
| Red    | Major              | `x.y.z → x+1.0.0`                  |
| Blue   | Compatible Blocked | Upgrade available but incompatible |
| Gray   | Ignored            | Skipped via config                 |

Non-color fallback:

* `[PATCH]`
* `[MINOR]`
* `[MAJOR]`
* `[BLOCKED]`

---

## 1.3 `latestx init` UX

### Flow

1. Workspace scan begins.
2. Detection summary displayed:

   * Language
   * Framework
   * Package manager
   * Manifest location
3. Confirmation step (if ambiguous).
4. `latestx.yaml` created.
5. Success summary.

### Edge Handling

* Multiple manifests detected → interactive selection.
* Mono-repo detected → list subprojects.

---

## 1.4 `latestx check`

### Default Behavior (TTY)

1. Display summary header:

   * Project type
   * Total dependencies
   * Upgradable count
   * Major/minor/patch breakdown

2. Render interactive table:

Columns:

* Selector checkbox
* Package name
* Current version
* Latest version
* Risk type
* Compatibility indicator
* Changelog link (if resolvable)

3. Keyboard Controls:

   * ↑ ↓ → navigation
   * Space → toggle select
   * A → select all safe (patch + minor)
   * M → select majors only
   * Enter → confirm update
   * Esc → cancel

---

### `--non-interactive`

* Prints table.
* No key interaction.
* Does not modify workspace.

---

### `--json`

Outputs structured JSON:

* metadata
* dependency list
* risk categories
* compatibility flags
* suggested safe upgrade set

---

## 1.5 `latestx update`

### Modes

| Mode              | Behavior                  |
| ----------------- | ------------------------- |
| Default           | Update selected or all    |
| `package-name`    | Single dependency         |
| `--dry-run`       | Simulation only           |
| `--compatibility` | AI-constrained update set |

### Update Execution Model

1. Create dependency snapshot.
2. Compute upgrade set.
3. Validate compatibility (if requested).
4. Call native package manager.
5. Validate installation success.
6. Show summary and diff.

# 2. `latestx.yaml` Schema Definition

## 2.1 Goals

* Deterministic reproducibility.
* Manual override support.
* Extensible for future features.

---

## 2.2 Top-Level Structure

```
version: 1

project:
  name:
  root:
  detected_at:

environment:
  language:
  framework:
  package_manager:
  manifest_path:
  lockfile_path:

preferences:
  allow_major_updates:
  interactive_default:
  enable_ai_compatibility:
  auto_select_safe_updates:

ignore:
  packages:
  patterns:

compatibility:
  strategy:
  strict_mode:

metadata:
  mono_repo:
  subprojects:
```

---

## 2.3 Field Definitions

### project

Descriptive metadata only.

### environment

Auto-detected fields. Editable but validated.

### preferences

Controls CLI behavior.

* `allow_major_updates`: boolean
* `enable_ai_compatibility`: boolean
* `auto_select_safe_updates`: boolean

### ignore

Explicit exclusions.

### compatibility

Defines resolution strategy:

* `strict` → No peer violations tolerated.
* `balanced` → Allow soft peer drift.
* `relaxed` → Only hard conflicts blocked.

# 3. Compatibility Decision Model

Compatibility is treated as a **constraint satisfaction + dependency graph problem**.

---

## 3.1 Inputs

* Dependency graph (direct + transitive).
* Peer dependency constraints.
* Framework-level constraints.
* Target versions.
* Lockfile state.

---

## 3.2 Constraint Types

### Hard Constraints

* Peer dependency upper/lower bounds.
* Engine version requirements.
* Framework version restrictions.

Violation → block upgrade.

---

### Soft Constraints

* Deprecated API warnings.
* Known ecosystem conflicts.
* Release notes breakage flags.

Violation → AI advisory only.

---

## 3.3 Resolution Algorithm (High-Level)

1. Parse current graph.
2. Identify candidate upgrades.
3. For each candidate:

   * Check semver compatibility.
   * Validate peer constraints.
   * Evaluate framework compatibility.
4. Build constraint matrix.
5. Solve:

   * Remove conflicting candidates.
   * Re-evaluate graph.
6. Produce maximal compatible upgrade set.

---

## 3.4 React Example Scenario

Upgrading from 16 → 18:

* Detect packages with peer constraint `react <17`.
* Query AI for known upgrade paths.
* Block incompatible libraries.
* Suggest:

  * Remove package
  * Downgrade React target
  * Upgrade dependent lib (if version exists)

---

## 3.5 Output Categories

| Status      | Meaning                      |
| ----------- | ---------------------------- |
| Safe        | No constraint violations     |
| Conditional | Requires coordinated upgrade |
| Blocked     | Hard conflict                |
| Unknown     | Insufficient metadata        |

# 4. AI Prompt and Guardrail Design

AI is advisory and analytical, never authoritative.

---

## 4.1 AI Responsibilities

* Analyze upgrade impact.
* Interpret ambiguous compatibility.
* Detect ecosystem upgrade patterns.
* Suggest coordinated version bumps.

---

## 4.2 AI Input Payload

Structured data only:

* Current dependency graph.
* Candidate upgrade graph.
* Peer dependency constraints.
* Framework metadata.
* Package release notes (if fetched).

---

## 4.3 AI Output Requirements

Structured JSON only:

* recommended_upgrade_set
* blocked_packages
* reasoning
* risk_assessment
* confidence_score

Free-text explanations allowed but must not override constraints.

---

## 4.4 Guardrails

1. AI cannot:

   * Execute commands.
   * Modify files.
   * Override hard semver violations.

2. Deterministic fallback:

   * If AI fails → revert to rule-based resolver.

3. Confidence threshold:

   * If confidence < defined threshold → warn user.

4. Rate limiting:

   * Prevent excessive token consumption.

---

## 4.5 Risk Management

If AI suggests:

* Removing critical package → require confirmation.
* Downgrading dependencies → explicit confirmation.
* Ignoring peer dependency → strict mode prevents.

# 5. Competitive Comparison

---

## 5.1 Renovate

Renovate

### Strengths

* Fully automated PR creation.
* Deep ecosystem support.
* CI-first design.

### Weaknesses

* Not interactive.
* No local CLI-first workflow.
* Heavy configuration complexity.

### Your Differentiation

* Local-first interactive control.
* AI compatibility mode.
* Designed for developer workflow, not only CI.

---

## 5.2 Dependabot

Dependabot

### Strengths

* Native GitHub integration.
* Automatic security patching.

### Weaknesses

* GitHub-only orientation.
* No real-time compatibility analysis.
* No interactive terminal UX.

### Differentiation

* Cross-platform.
* AI-assisted compatibility graph reasoning.
* Not repository-host dependent.

---

## 5.3 npm-check-updates

npm-check-updates

### Strengths

* Simple CLI.
* Fast.
* JS ecosystem focused.

### Weaknesses

* JavaScript only.
* No compatibility intelligence.
* No cross-ecosystem support.
* No AI assistance.

### Differentiation

* Multi-language.
* Cross-ecosystem.
* Compatibility mode.
* Interactive UI.
